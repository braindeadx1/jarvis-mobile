1. Uebersicht

Jarvis ist ein persoenlicher KI-Sprachassistent bestehend aus:

  • Android PWA fuer Handy-Steuerung
  • ReSpeaker Lite Voice Satellite mit „Hey Jarvis“ Wake Word
  • FastAPI Backend auf Proxmox CT
  • Home Assistant Integration fuer Smarthome
  • ClawBot fuer WhatsApp/Discord Nachrichtenweiterleitung

3. Komponenten im Detail

HW ReSpeaker Lite (Voice Satellite)

  • Hardware: Seeed ReSpeaker Lite, ESP32-S3 + XMOS XU316 DSP
  • Firmware: ESPHome 2026.3.3 (standard i2s_audio, kein Custom Component)
  • I2S Config: bits_per_sample 32bit, i2s_mode secondary (XMOS ist Master)
  • Pins: LRCLK=GPIO7, BCLK=GPIO8, MCLK=GPIO9, DIN=GPIO44, DOUT=GPIO43, LED=GPIO1
  • Anschluss: USB am NUC (192.167.178.11) fuer Strom + Serial Flash
  • WiFi: ESPHome API zu Home Assistant
  • LED: WS2812 — Cyan=Listening, Blau=Processing, Rot=Error
  • Wake Word: server-side via openWakeWord (hey_jarvis Modell)
  • Speaker: noch nicht aktiv (I2S Bus-Lock Problem, Full-Duplex WIP)

HA HA Voice Pipeline „Jarvis“

  • Wake Word: openWakeWord Add-on, Modell hey_jarvis, Threshold 0.5
  • STT: Whisper (faster-whisper), deutsch
  • Conversation Engine: Custom Integration jarvis_conversation
    • Leitet STT-Text per REST an Jarvis Server
    • Endpoint: POST https://192.167.200.30:8443/v1/chat/completions
    • OpenAI-kompatibles Format
  • TTS: Piper (de_DE-eva_k-x_low)

SRV Jarvis Server (CT 7300)

  • Stack: FastAPI + Uvicorn, Python venv
  • Port: 8443 (HTTPS, selbstsigniert)
  • LLM: OpenRouter (GPT-4o-mini, konfigurierbar)
  • TTS: Edge TTS (de-DE-ConradNeural)
  • HA Integration: REST API fuer Smarthome-Steuerung
  • Geraete: ~90 Geraete in devices.csv (hot-reload)
  • WebSocket: fuer PWA-Clients (Audio + Text broadcast)
  • Endpoints: /ws (WebSocket), /webhook/clawbot, /api/voice, /v1/chat/completions

BOT Friday Bridge (CT 7200)

  • Typ: Python Script, ueberwacht ClawBot Session-JSONL-Logs
  • Erkennung: Bot-Antworten (role: assistant)
  • Filterung: Strippt final-Tags, filtert NO_REPLY / HEARTBEAT_OK
  • Smarthome-Meldungen – Cooldown-System:
    • Wasser = 0 (immer sofort)
    • Klingel = 30 s
    • Person = 5 min
    • Fenster = 5 min
    • Temperatur = 30 min
  • Weiterleitung: per Webhook an Jarvis Server
  • Fix 2026-04-14: Session-Rotation nach 30 s Idle (war vorher blockiert)

4. Sprachbefehl-Flow

  1. Mikrofon nimmt Audio auf (I2S, XMOS DSP)
  2. Audio-Stream per WiFi an HA (ESPHome API, continuous)
  3. openWakeWord prueft auf „Hey Jarvis“
  4. Wake Word erkannt → Pipeline startet, LED cyan
  5. Whisper STT transkribiert Sprachbefehl
  6. LED blau, Text an Jarvis Conversation Agent
  7. Custom Integration sendet REST POST an Jarvis Server
  8. Jarvis verarbeitet: parse_ha_command() → HA-Steuerung ODER LLM → OpenRouter
  9. Antwort zurueck an HA Pipeline + Broadcast an PWA
  10. Piper TTS fuer lokale Ausgabe (Speaker WIP)
  11. PWA empfaengt Antwort per WebSocket + Edge TTS Audio
  12. Pipeline fertig, zurueck zu Wake Word Listening

6. Bekannte Einschraenkungen

  • Speaker auf ReSpeaker nicht aktiv: ESPHome I2S Bus Mutex Lock verhindert gleichzeitige Mic+Speaker Nutzung. ESP-IDF v5.x unterstuetzt Full-Duplex ueber separate RX/TX Channels, aber ESPHome’s Lock ist der Blocker.
  • Custom Component QingWind6: inkompatibel mit ESPHome 2026.3.3 (API-Aenderungen)
  • openWakeWord Wake Word Entity: zeigt in HA „unavailable“ — kosmetisch, Erkennung funktioniert server-side
  • Wake Word Threshold: ggf. anpassen je nach Umgebungslaerm
